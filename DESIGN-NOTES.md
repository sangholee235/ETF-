# 설계 결정 메모 (아키텍처 회의 결론)

전략 관리·배포·패키징을 두고 나눈 논의의 결론. "지금 뭘 하고, 뭘 보류하는지"의 기준.

---

## 1. 전략은 어디에 사나 — 파일이 진실원천
- 전략은 `bot_config_{broker}.json` **파일**에 있고, 스케줄러가 **매 tick 마다 파일을 새로 읽어** 매수(`run_once → BotConfig.load()`).
- 즉 **hot-reload 가 이미 됨** — 폼으로 고치든 파일을 직접 고치든 다음 tick 에 반영. 재시작 불필요.
- 대시보드 폼 = 그 파일에 쓰는 편집기(`PATCH /api/bot/config`). "프론트가 전략 고쳐 백엔드에 올린다"는 이미 동작 중.

## 2. 결정 — 심플 번들, 셀프호스팅, 각자 사용 (현재 채택)
- **프론트+백엔드 한 덩어리**(docker compose)로 배포, 각자 자기 서버에서 돌리고 자기가 씀.
- 폼이 런타임에 전략을 바로 바꿈 = **편의 우선.** "런타임 봉인"은 지금은 없음(각자 자기 기계·자기만 쓰니 불필요).
- 각자 자기 키·자기 전략·자기 거래 → **법적으로 안전한 셀프호스팅 모델.**
- 키 입력은 **각자 자기 번들의 프론트 → 자기 백엔드(localhost)** 로. (내가 중앙 호스팅한 페이지에 남의 키 입력하는 구조는 ❌ — mixed-content 차단 + 책임/노출)

## 3. 보류 — "방진시설"(GitOps/불변 인프라) 모드
동기: 클라우드에 올렸을 때 **사용자든 아니든 런타임에 아무도 전략을 못 바꾸게**, 변경은 오직 git→CI/CD 로만. 감사 가능.
결론: **좋은 방향이지만 지금은 안 만든다.** 나중에 "방진 모드"로 옵트인. 설계도만 남겨둠:

- **전략 = 공개 git repo** (키 같은 비밀 없으니 공개 OK, 자격증명 0). 변경은 로컬 commit→push 만.
- **비밀(키) = git 밖.** 배포 시 env/시크릿 주입. 전략(공개)과 키(비밀) 물리 분리.
- **전달 = B-pull** — 프로덕션이 **스스로 주기적으로 git pull**(아웃바운드만, 인바운드 0). 폴링은 Flux(1분)/ArgoCD(3분) 처럼 GitOps 정석. 적립봇은 하루 1번 사니 **매수 직전 1회 pull 로도 충분**(비용 무시 가능).
  - puller 신원 = **읽기전용, repo 1개 스코프**(deploy key) 또는 공개면 무자격. 박스가 털려도 소스 못 더럽힘.
- **검증** = CI 에서 스키마검증(깨진 전략은 배포 차단) + 박스 2차검증 + 라스트굿 폴백.
- **프로덕션 봉인** = GET 만. 전략변경 엔드포인트 0개.
- **정지의 비대칭** — 정지(stop)는 fail-safe(매수전용이라 안 사면 무해), 변경(mutate)은 fail-dangerous. 그래서 **변경은 git 만, 정지는 운영평면**(클라우드 콘솔 인스턴스 정지 / SSH `docker stop`). 정지 통로가 열려도 최악이 "멈춤"이라 봉인 안 깨짐.
- **폼** = 봉인 모드에선 프로덕션 직접 편집 대신 **로컬 작성도우미**(출력이 커밋). 편의 유지 + git 게이트.

## 4. 두 모드, 하나의 코어 (미래 그림)
| | 심플 모드 (기본·현재) | 방진 모드 (옵트인·보류) |
|---|---|---|
| 대상 | 처음 쓰는 사람 | 보안·감사 진심 |
| 설치 | `docker compose up` | + git repo + puller |
| 전략 편집 | 웹 폼이 파일에 바로 씀 | 로컬 작성 → commit → pull |
| 쓰기 엔드포인트 | 켜짐 | 꺼짐(봉인) |
| 키 입력 | 폼(온보딩 마법사) | env/시크릿 주입 |
- 차이는 딱 하나: **"전략 파일을 누가 쓸 수 있나"**(로컬 API vs git-pull). 같은 코어 + 모드 플래그.

## 5. 패키징 — 설치형 앱? (검토, 미결)
- 가능. 가장 가벼운 건 **PyInstaller 단일 exe**(FastAPI 가 정적 프론트까지 서빙 + 브라우저 자동 오픈). 도커 0.
- 진짜 창 원하면 Tauri/Electron(파이썬 백엔드 자식 프로세스 번들 — 손 더 감).
- **함정: 데일리 스케줄 봇은 매수 시각에 기계가 켜져 있어야 함.** 노트북 설치형은 놓칠 수 있음 → 자동실행+트레이+놓침보정(catch-up) 필요.
- 그래서 "넣고 잊는" 자동 적립엔 **항상 켜진 서버(현재 Lightsail)가 더 신뢰적.** 설치형은 온보딩만 쉬움.
- 결론: 지금은 **서버 모델 유지.** 설치형은 나중 옵션.

## 6.5. 미결 — 매수 주문을 시장가로 할지, 지정가(+프리미엄)로 할지 (2026-07-06)
그리디 다중매수(하루 예산을 그리디 루프로 여러 ETF에 순차 배분)를 실제 LIVE 계좌에
적용하면서 **시장가 주문이 "매수증거금 부족"으로 거부**되는 문제 발견.

- **원인(가설, 미확정)**: 시장가는 체결가가 상한가(+30%)까지 튈 수 있어 증권사가
  그 기준으로 증거금을 미리 예약함. 현재가만 보고 계산하면 실제 필요액을 과소평가.
- **시도했다 되돌린 것**: `kt00010`(주문인출가능금액요청)의 `profa_100ord_alowq`로
  주문 직전 실제가능수량을 재확인해 클램프 — **정상 계좌의 매수까지 잘못 막는
  부작용**으로 되돌림(그 필드가 신용거래 맥락일 가능성, 미확인).
- **현재 임시 조치**: 수량 계산에 추측 버퍼(현재가×1.3)를 곱해 보수적으로 계획.
  **부작용**: 버퍼가 실제보다 작게 계산해 한 번에 살 수 있던 수량이 여러 번의
  소액 매수로 쪼개짐(비효율, 로그 지저분). 사용자가 "이거 버그같다" 지적.
- **제안된 대안(미구현, 논의만 됨)**: 시장가 대신 **"현재가+소폭 프리미엄(예 0.5%)
  지정가"**로 주문. 지정가는 정확한 가격을 아니까 상한가 버퍼가 필요 없어 자금을
  정확하게 씀. 프리미엄만큼 매도 대기 물량과 바로 만나 사실상 즉시 체결도 노림.
  `config.round_up_to_tick` 헬퍼(올림 버전, 기존 `round_to_tick`은 내림이라 프리미엄
  붙인 가격엔 안 맞음)까지 설계했으나 **구현 전 사용자가 "확정 없이 코드부터
  고치지 말라"고 중단시켜 커밋 없이 되돌림.**
- **다음 세션 할 일**: 이 지정가+프리미엄 전환을 사용자와 먼저 확정하고 진행할 것.
  확정되면: `portfolio.plan_daily_buys`에서 버퍼 곱하기 대신 `round_up_to_tick(가격×(1+프리미엄), tick)`으로
  주문가를 계산하고, `runner.py`의 Decision을 `MARKET_BUY`→`LIMIT_BUY`(price=계산된 지정가)로 변경.

## 6. 자원 판단 (배포 사양)
- 런타임은 I/O 대기 위주라 CPU 거의 0, 하루 1 tick 만 잠깐. **1GB 로 충분.**
- 유일한 관건은 **프론트 빌드 메모리**(`npm build` 스파이크) → 스왑 2GB 또는 빌드는 로컬/CI.
- 디스크·전송은 압도적 여유.
